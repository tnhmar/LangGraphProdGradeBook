\begin{learningbox}
After completing this chapter you will be able to:
\begin{itemize}
  \item Separate \term{resource ceilings} from \term{semantic-quality guards}
        by placing them in two distinct frozen configuration objects,
        each with its own update cadence and override surface.
  \item Implement a \term{two-layer guard} whose outer layer is
        LangGraph's \code{recursion\_limit} and whose inner layer is
        an application-level check that fires \emph{before} the outer
        layer is ever needed.
  \item Detect \term{stagnation} using a structural SHA-256 hash of the
        goal stack rather than embedding similarity, eliminating the extra
        model call that cosine-distance approaches require.
  \item Detect \term{stuck-state} runs --- cycles that produce no valid
        decision --- and route them to a recovery node without aborting
        the run.
  \item Enforce a \term{delegation-depth limit} that prevents infinite
        chains of agent-to-agent calls in multi-agent topologies
        introduced in Chapters~17 and~19.
\end{itemize}
\end{learningbox}

\section{Two Kinds of Stopping Condition}
\label{sec:two-kinds}
% ─────────────────────────────────────────────────────────────────────────────

Volume~1 Chapter~2 §2.5--2.7 identifies two families of stopping condition
that must not be conflated.

\term{Resource ceilings} are hard limits on consumption: maximum cycles,
token budget, wall-clock time, and cost in USD.  They are set once per
run by the operator and do not change.  When a resource ceiling fires, the
run terminates with a specific \code{TerminationReason} --- there is
nothing to recover from.

\term{Semantic-quality guards} detect when the agent's behaviour has
degraded without exhausting any resource.  A run may be well within its
token budget yet making no progress because the goal stack has not changed
shape for five cycles.  These guards fire into \emph{recovery nodes}, not
directly into termination, because the degraded pattern may be correctable.

Placing both families in a single configuration object creates a collision:
operators who want to increase the stuck-state window without touching the
cost cap must reason about an object that mixes concerns.  This chapter
uses two separate frozen models: \component{TerminationPolicy} for resource
ceilings and \component{GuardConfig} for semantic guards.

% ─────────────────────────────────────────────────────────────────────────────
\section{TerminationPolicy and GuardConfig}
\label{sec:configs}
% ─────────────────────────────────────────────────────────────────────────────

Both objects are \code{frozen=True} Pydantic models.  Frozen models
serialise cleanly to JSON (LangGraph checkpoints persist them into
\code{AgentState.\_policy} and \code{AgentState.\_guard}), and they
cannot be mutated inside a running graph.

\begin{lstlisting}[language=Python,
    caption={TerminationPolicy and GuardConfig --- two frozen config models
             (src/state.py)},
    label={lst:term-policy}]
class TerminationPolicy(BaseModel):
    """Resource ceilings: set once per run, never recovered from."""
    model_config = ConfigDict(frozen=True)
    max_cycles:         int   = Field(default=20,      ge=1,  le=500)
    token_ceiling:      int   = Field(default=100_000, ge=1_000)
    wall_clock_timeout: float = Field(default=300.0,   ge=1.0)  # seconds
    cost_cap:           float | None = None             # USD; None = unlimited
    max_retries:        int   = Field(default=3,       ge=1, le=10)
    max_chain_depth:    int   = Field(default=5,       ge=1)

class GuardConfig(BaseModel):
    """Semantic-quality guards: recoverable; separate update cadence."""
    model_config = ConfigDict(frozen=True)
    stuck_state_window:   int   = Field(default=3,    ge=1)
    stagnation_window:    int   = Field(default=3,    ge=1)
    divergence_threshold: float = Field(default=0.25, ge=0.0, le=1.0)
\end{lstlisting}
\githubref{src.state.TerminationPolicy}

\medskip
Both objects are passed into \code{build\_agent\_graph()} and serialised
into \code{AgentState.\_policy} and \code{AgentState.\_guard} by
\code{make\_initial\_state()}.  Every guard function reads from
\code{state.get("\_policy")} and \code{state.get("\_guard")} rather than
from a global singleton, so different runs in the same process can carry
different policies without interference.

\begin{notebox}
\textbf{\code{max\_retries} is shared deliberately.}
\code{TerminationPolicy.max\_retries} is the single source of truth used
by Instructor's parse-repair loop (Chapter~4), the Tenacity retry
decorator (Chapter~5), and the post-action plan re-validation step in the
review subgraph (Chapter~1 §\ref{sec:review-subgraph}).  Duplicating it
into a separate field on each component creates drift.
\end{notebox}

\section{The Two-Layer Guard Architecture}
\label{sec:two-layer}
% ─────────────────────────────────────────────────────────────────────────────

LangGraph enforces a hard iteration ceiling through \code{recursion\_limit}
in \code{RunnableConfig}.  When the graph exceeds this ceiling,
\code{GraphRecursionError} is raised at the runtime level.
Chapter~1 §\ref{sec:graph-wiring} set \code{recursion\_limit = max\_cycles
$\times$ 4} to leave room for recovery branches.

This ceiling is Layer~2.  It exists to catch bugs --- a misconfigured
recovery branch that loops, or a node that forgets to advance
\code{run\_state}.  It should never fire in a correctly written agent.

Layer~1 is the application-level check in \code{pre\_cycle\_gate()}, which
runs at the top of each cycle before the perception subgraph is entered.
It fires \emph{earlier} and \emph{more informatively} than
\code{GraphRecursionError}: it sets a \code{TerminationReason}, writes a
structured log entry, and exits cleanly through a named edge rather than
raising an exception.

\begin{warningbox}
\textbf{Never use \code{recursion\_limit} as the primary guard.}
\code{GraphRecursionError} carries no \code{TerminationReason}, no
\code{dead\_letter} snapshot, and no Langfuse event.  Operators inspecting
a failed run in the dashboard will see a stack trace rather than a
structured cause.  Layer~1 must always fire first.
\end{warningbox}

\section{The \texttt{pre\_cycle\_gate} Function}
\label{sec:pre-cycle-gate}
% ─────────────────────────────────────────────────────────────────────────────

\code{pre\_cycle\_gate()} is a conditional edge function wired between the
\code{reflect} node and the \code{perceive} subgraph.  It returns a routing
string: \code{"continue"}, \code{"terminal"}, \code{"suspend"},
\code{"stagnation\_recovery"}, or \code{"stuck\_recovery"}.  No state is
written here --- the function only reads.

\begin{lstlisting}[language=Python,
    caption={\texttt{pre\_cycle\_gate} --- Layer-1 guard routing (src/guards.py)},
    label={lst:pre-cycle-gate}]
def pre_cycle_gate(state: AgentState, config: RunnableConfig) -> str:
    policy_dict = state.get("_policy") or {}
    guard_dict  = state.get("_guard")  or {}

    try:
        policy = TerminationPolicy(**policy_dict)
        guard  = GuardConfig(**guard_dict)
    except Exception:
        policy = TerminationPolicy()
        guard  = GuardConfig()

    run_state = state.get("run_state", RunState.PLANNING)

    if isinstance(run_state, RunState) and run_state.is_terminal():
        return "terminal"
    if run_state == RunState.SUSPENDED_HITL:
        return "suspend"

    # Resource ceilings
    if (state.get("cycle_count") or 0) >= policy.max_cycles:
        return "terminal"
    budget = state.get("token_budget")
    if budget and budget.is_exhausted():
        return "terminal"

    # Semantic-quality guards
    if check_stagnation(state, guard):
        return "stagnation_recovery"
    if check_stuck(state, guard):
        return "stuck_recovery"

    return "continue"
\end{lstlisting}
\githubref{src.guards.pre\_cycle\_gate}

The ordering inside \code{pre\_cycle\_gate} matters.  Terminal run states
are checked first because they are set by \emph{previous} review or
reflection steps and must not be overridden.  Resource ceilings come second
because they are hard constraints.  Semantic guards come last because they
route to recovery, not termination.

\section{N1: The Stagnation Detector}
\label{sec:stagnation}
% ─────────────────────────────────────────────────────────────────────────────

The stagnation detector asks whether the agent has made any structural
progress on its goal stack.  It does this by hashing the current goal-stack
shape and comparing that hash to the preceding \code{stagnation\_window}
hashes.  If they are all identical, the run has stagnated.

\subsection{Goal-state hashing}
\label{subsec:goal-hash}

The hash function operates on the \emph{structure} of the goal stack
(identifiers, statuses, and descriptions), not on beliefs or working
memory.  This is the framework's approach (Java
\code{DefaultLivenessDetector.hashGoalState()}): structural progress is
visible in goal-status transitions, and those transitions are what the
agent should be driving.

\begin{lstlisting}[language=Python,
    caption={\texttt{hash\_goal\_state} --- structural hash of the goal stack
             (src/guards.py)},
    label={lst:hash-goal}]
def hash_goal_state(goals: list[Goal]) -> str:
    """
    SHA-256 of the sorted goal-stack structure.
    Status transitions (PENDING -> ACTIVE -> COMPLETED) change the hash.
    Text changes that do not change status do NOT progress the agent.
    Returns the first 16 hex characters — sufficient for collision-free
    comparison across the stagnation_window.
    """
    snapshot = json.dumps(
        [
            {
                "id":          g.id,
                "status":      g.status.value,
                "description": g.description,
            }
            for g in sorted(goals, key=lambda g: g.id)
        ],
        sort_keys=True,
    )
    return hashlib.sha256(snapshot.encode()).hexdigest()[:16]
\end{lstlisting}
\githubref{src.guards.hash\_goal\_state}

\begin{notebox}
\textbf{Why not use embedding cosine distance?}
Cosine distance between the original goal embedding and the current goal
summary requires an extra model call per cycle and a stored embedding
vector.  It answers a different question (``has the goal drifted from its
original intent?'') than the stagnation detector (``has any progress
occurred?'').  Volume~1 §2.6 specifies stagnation as a structural check.
If semantic drift detection is required, add it as a separate \emph{fifth}
check in \code{pre\_cycle\_gate} with its own recovery path.
\end{notebox}

\subsection{Stagnation check and recovery}

\code{check\_stagnation()} reads \code{goal\_hash\_history} from state and
returns \code{True} when the last \code{stagnation\_window} hashes are
identical.  The hash is appended to the history at the end of each
\code{reflect\_node} call:

\begin{lstlisting}[language=Python,
    caption={Stagnation check and recovery node (src/guards.py)},
    label={lst:stagnation}]
def check_stagnation(state: AgentState, guard: GuardConfig) -> bool:
    history = state.get("goal_hash_history") or []
    w       = guard.stagnation_window
    if len(history) < w:
        return False
    return len(set(history[-w:])) == 1      # all identical

@observe(name="stagnation_recovery_node")
def stagnation_recovery_node(
    state: AgentState, config: RunnableConfig
) -> dict:
    """Reset plan state; force a fresh perception cycle."""
    return {
        "revision_count":    0,
        "goal_hash_history": [],   # clear — avoids immediate re-trigger
        "decision":          None,
        "reflect_output":    None,
        "run_state":         RunState.PLANNING,
    }
\end{lstlisting}
\githubref{src.guards.check\_stagnation}

The recovery node clears \code{goal\_hash\_history} so it does not
immediately re-trigger on the next cycle.  It also clears \code{decision}
and \code{reflect\_output} so the agent enters the next cycle without stale
outputs from the stuck cycle.

\section{N2: The Stuck-State Detector}
\label{sec:stuck}
% ─────────────────────────────────────────────────────────────────────────────

Stagnation measures goal-stack progress.  Stuck-state measures
\emph{decision-layer} progress: has the agent produced a valid
\code{ToolRequest} or \code{FinalAnswer} in the last
\code{stuck\_state\_window} cycles?  A cycle whose \code{ReflectNode}
did not emit a verdict (because the agent produced only
\code{ReasoningTrace} outputs that Instructor could not resolve) will have
\code{verdict = None} in its \code{CycleTrace}.

\begin{lstlisting}[language=Python,
    caption={Stuck-state detector and recovery node (src/guards.py)},
    label={lst:stuck}]
def check_stuck(state: AgentState, guard: GuardConfig) -> bool:
    traces = state.get("cycle_traces") or []
    w      = guard.stuck_state_window
    if len(traces) < w:
        return False
    recent_verdicts = [t.verdict for t in traces[-w:]]
    return all(v is None for v in recent_verdicts)

@observe(name="stuck_recovery_node")
def stuck_recovery_node(
    state: AgentState, config: RunnableConfig
) -> dict:
    policy_dict   = state.get("_policy") or {}
    revision_count = (state.get("revision_count") or 0) + 1
    max_retries   = policy_dict.get("max_retries", 3)

    if revision_count >= max_retries:
        return {
            "run_state":         RunState.TERMINATED,
            "termination_reason": TerminationReason.STUCK_STATE,
        }
    disambiguation = WorkingMemoryEntry(
        content = (
            f"[RECOVERY cycle={state.get('cycle_count',0)}] "
            "Previous cycles produced no valid decision. "
            "Try a different approach or ask for clarification."
        ),
        tier  = WorkingMemoryTier.ACTIVE,
        trust = TrustTier.HIGH,
    )
    return {
        "working_memory":  [disambiguation],
        "revision_count":  revision_count,
        "decision":        None,
        "run_state":       RunState.PLANNING,
    }
\end{lstlisting}
\githubref{src.guards.check\_stuck}

\medskip
The recovery node injects a disambiguation entry into working memory at
\code{WorkingMemoryTier.ACTIVE} --- the tier that is never evicted
(Chapter~7) --- so it is guaranteed to reach the \code{reason\_node} on
the next cycle.  If \code{revision\_count} reaches \code{max\_retries},
the recovery node terminates with \code{STUCK\_STATE} rather than looping
indefinitely.

\begin{tipbox}
\textbf{Two detectors, one shared threshold.}
Both \code{stagnation\_window} and \code{stuck\_state\_window} default to
\code{3} in \code{GuardConfig}.  They are separate fields rather than a
single \code{N} parameter because they measure different things and may
legitimately differ.  A long-horizon planning agent might raise
\code{stagnation\_window} to 5 while keeping \code{stuck\_state\_window}
at 3.
\end{tipbox}

\section{The Delegation-Depth Guard}
\label{sec:delegation-depth}
% ─────────────────────────────────────────────────────────────────────────────

When an agent spawns a sub-agent (Chapter~17 for A2A, Chapter~19 for the
supervisor pattern), the sub-agent increments \code{AgentState.chain\_depth}
by one before beginning its own cycles.  Without a ceiling on this value,
a buggy delegation chain can spawn sub-agents indefinitely.

\code{TerminationPolicy.max\_chain\_depth} (default 5) sets that ceiling.
The guard runs in \code{delegation\_depth\_guard\_node}, which is wired
at the \code{INITIALIZED} state --- before any perception or reasoning
occurs.

\begin{lstlisting}[language=Python,
    caption={Delegation-depth guard (src/guards.py)},
    label={lst:delegation-depth}]
@observe(name="delegation_depth_guard_node")
def delegation_depth_guard_node(
    state: AgentState, config: RunnableConfig
) -> dict:
    """
    Called at INITIALIZED before any cycle begins.
    A sub-agent whose chain_depth >= max_chain_depth refuses the task.
    """
    policy_dict = state.get("_policy") or {}
    max_depth   = policy_dict.get("max_chain_depth", 5)
    current     = state.get("chain_depth") or 0

    if current >= max_depth:
        langfuse_context.update_current_observation(metadata={
            "delegation_depth_exceeded": True,
            "depth": current,
        })
        return {
            "run_state":          RunState.TERMINATED,
            "termination_reason": TerminationReason.DELEGATION_DEPTH,
            "dead_letter": {
                "error": f"DELEGATION_DEPTH_EXCEEDED (depth={current})",
                "cycle": 0,
            },
        }
    return {"run_state": RunState.VALIDATING}
\end{lstlisting}
\githubref{src.guards.delegation\_depth\_guard\_node}

The guard's return value for the success path (\code{RunState.VALIDATING})
triggers resource pre-checks --- token budget validation, policy load ---
before the first perception cycle.  This mirrors the Java framework's
\code{StateMachineRunner.step(INITIALIZED)} sequence exactly.

\section{Wiring the Guards into the Graph}
\label{sec:guard-wiring}
% ─────────────────────────────────────────────────────────────────────────────

The guards integrate into the outer graph from Chapter~1 through three
additions: the \code{delegation\_depth\_guard\_node} at the entry point,
the \code{pre\_cycle\_gate} conditional edge between \code{reflect} and
\code{perceive}, and two recovery nodes.

\begin{lstlisting}[language=Python,
    caption={Guard nodes added to the outer graph (src/graph.py additions)},
    label={lst:guard-wiring}]
# In build_agent_graph(), add to StateGraph g:
g.add_node("depth_guard",          delegation_depth_guard_node)
g.add_node("stagnation_recovery",  stagnation_recovery_node)
g.add_node("stuck_recovery",       stuck_recovery_node)

# Entry point: depth check before first cycle
g.add_edge(START, "depth_guard")
g.add_conditional_edges("depth_guard", _route_after_depth, {
    "validating": "perceive",  # passes depth check
    "terminal":   END,
})

# Between reflect and next perceive cycle
g.add_conditional_edges("reflect", route_reflect_verdict, {
    "perceive":            "perceive",
    "reason":              "reason",
    "escalation":          "escalation",
    "terminal":            END,
    "stagnation_recovery": "stagnation_recovery",
    "stuck_recovery":      "stuck_recovery",
})

# Recovery nodes loop back to perceive
g.add_edge("stagnation_recovery", "perceive")
g.add_edge("stuck_recovery",      "perceive")
\end{lstlisting}
\githubref{src.graph.build\_agent\_graph}

\medskip

Routing the recovery nodes \emph{back to} \code{perceive} rather than
\code{reason} is deliberate.  The recovery node writes a disambiguation
entry to working memory.  That entry must pass through the perception
pipeline's relevance filter and grounding stages before it reaches the
reasoning node, ensuring it is trust-annotated and budget-managed like
any other input.

\begin{warningbox}
\textbf{Recovery nodes can themselves trigger guards.}
If \code{stagnation\_recovery\_node} runs but the agent stagnates again
after three more cycles, \code{pre\_cycle\_gate} will route to
\code{stagnation\_recovery} again.  The second pass increments
\code{revision\_count}.  When \code{revision\_count $\geq$ max\_retries},
the recovery node transitions to \code{RunState.TERMINATED} with
\code{STUCK\_STATE} rather than routing back to perceive.  Without this
ceiling, a recovery node can become the source of the infinite loop it
was meant to cure.
\end{warningbox}

\section{GraphRecursionError: the Outer Net}
\label{sec:recursion-error}
% ─────────────────────────────────────────────────────────────────────────────

If a bug causes the graph to iterate past \code{max\_cycles $\times$ 4},
LangGraph raises \code{GraphRecursionError}.  The \code{run\_agent()}
function in \code{src/graph.py} catches it at the invocation site and
converts it into a \code{dead\_letter} record:

\begin{lstlisting}[language=Python,
    caption={Catching GraphRecursionError at the runner level (src/graph.py)},
    label={lst:recursion-error}]
from langgraph.errors import GraphRecursionError

async def run_agent(
    graph:  CompiledStateGraph,
    state:  AgentState,
    policy: TerminationPolicy,
    guard:  GuardConfig,
    agent_id: str,
    session_id: str | None = None,
) -> AgentState:
    run_config = RunnableConfig(
        recursion_limit = policy.max_cycles * 4,
        callbacks       = [LangfuseCallbackHandler(session_id=session_id)],
        configurable    = {
            "thread_id": state.get("run_id", agent_id),
            "policy":    policy.model_dump(),
            "guard":     guard.model_dump(),
        },
    )
    try:
        return await graph.ainvoke(state, run_config)
    except GraphRecursionError as exc:
        return {
            **state,
            "run_state":          RunState.TERMINATED,
            "termination_reason": TerminationReason.MAX_CYCLES,
            "dead_letter": {
                "error": str(exc),
                "cycle": state.get("cycle_count", 0),
            },
        }
\end{lstlisting}
\githubref{src.graph.run\_agent}

The catch block merges the original state with the terminal fields rather
than returning a fresh dict.  This preserves the last checkpoint's belief
state and working memory for post-mortem inspection even when the outer
runtime ceiling fires.

\begin{takeawaybox}[Chapter~2 Key Takeaways]
\begin{itemize}
  \item \textbf{Two config objects, two concerns.}
        \component{TerminationPolicy} owns resource ceilings (hard, no
        recovery).  \component{GuardConfig} owns semantic-quality thresholds
        (soft, recovery nodes route back to \code{perceive}).

  \item \textbf{Two-layer guard: app then runtime.}
        \code{pre\_cycle\_gate()} fires at Layer~1 with a structured
        \code{TerminationReason}.  \code{recursion\_limit} is Layer~2 and
        catches only programming errors.  Layer~1 must always fire first.

  \item \textbf{Stagnation is structural, not semantic.}
        SHA-256 of the sorted goal stack detects when no goal-status
        transitions have occurred in \code{stagnation\_window} cycles.
        No embedding call required.

  \item \textbf{Stuck-state detects decision-layer failure.}
        Consecutive \code{CycleTrace.verdict = None} entries mean the agent
        is reasoning without reaching a \code{ToolRequest} or
        \code{FinalAnswer}.  The recovery node injects a
        \code{WorkingMemoryTier.ACTIVE} disambiguation entry.

  \item \textbf{Recovery nodes need their own ceiling.}
        Without checking \code{revision\_count $\geq$ max\_retries}, a
        recovery node can itself cause the infinite loop it was written to
        prevent.

  \item \textbf{Delegation depth protects multi-agent topologies.}
        \code{TerminationPolicy.max\_chain\_depth} (default 5) is checked
        at \code{INITIALIZED}, before the first perception cycle, so no
        resources are consumed before a deep chain is refused.
\end{itemize}
\end{takeawaybox}

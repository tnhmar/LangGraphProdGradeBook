\begin{learningbox}
After completing this chapter you will be able to:
\begin{itemize}
  \item Distinguish \term{output-schema validation} (is the JSON
        well-formed?) from \term{plan-coherence validation} (does the
        decision make sense given the goal stack?) and place each at the
        correct point in the graph.
  \item Select an Instructor mode --- \code{TOOLS}, \code{JSON}, or
        \code{MD\_JSON} --- based on what a given model provider actually
        supports, rather than defaulting blindly.
  \item Implement a bounded repair loop that escalates rather than
        retrying forever when a plan is structurally incoherent.
  \item Apply the Subgraph Principle to \component{reason\_node} and
        confirm, by the same four-point test used throughout this book,
        that it correctly remains a single node.
\end{itemize}
\end{learningbox}

\section{Three Validation Tiers, Three Locations}
\label{sec:three-tiers}
% ─────────────────────────────────────────────────────────────────────────────

Three genuinely different validation questions are asked at three
genuinely different points in the agent's cycle.  Volume~1 §3.x is
explicit that collapsing them produces validators that run either too
early or too late.

\begin{table}[htbp]
\centering
\caption{The three validation tiers and where each lives}
\label{tab:validation-tiers}
\renewcommand{\arraystretch}{1.5}
\begin{tabularx}{\linewidth}{@{} l l P{5.2cm} @{}}
\toprule
\textbf{Tier} & \textbf{Where} & \textbf{Question} \\
\midrule
0. Output schema
   & Inside \code{reason\_node},
     handled by Instructor internally
   & Is the LLM's raw response parseable into the declared
     \code{Decision} type? \\
0.5 Plan coherence
   & Inside \code{reason\_node},
     after Instructor returns
   & Given the current \code{GoalStack}, does this specific decision
     make sense? \\
1--4 Action validation
   & Inside \code{action\_node}
     (Chapter~1 §\ref{sec:action-node})
   & Is this specific tool call, with these specific arguments, safe and
     authorised to dispatch \emph{right now}? \\
\bottomrule
\end{tabularx}
\end{table}

This chapter covers Tier~0 and Tier~0.5.  Tier~0 asks \emph{can this be
parsed at all}, independent of the agent's goals.  Tier~0.5 asks
\emph{should this decision happen}, which requires the \code{GoalStack}
and is only meaningful once parsing has already succeeded.

\section{Tier 0: Output Schema Validation with Instructor}
\label{sec:instructor}
% ─────────────────────────────────────────────────────────────────────────────

Every \code{Decision} subtype --- \code{ToolRequest}, \code{FinalAnswer},
\code{Escalate}, \code{AskClarification}, \code{ParallelToolRequests} ---
is a strict Pydantic model with \code{strict=True} (Chapter~1).
Instructor wraps the LiteLLM client so the model call returns one of
these types directly, re-prompting the model with the validation error
when the first attempt does not parse.

\begin{lstlisting}[language=Python,
    caption={Instructor-wrapped decision call (src/graph.py, excerpt)},
    label={lst:instructor-call}]
client   = instructor.from_litellm(litellm.completion)
decision: Decision = client.chat.completions.create(
    model          = model,
    response_model = Decision,
    messages       = [
        {"role": "system", "content": system_prompt + decision_schema},
        {"role": "user",   "content": f"Current goal: {goal}"},
    ],
    max_retries = policy.max_retries,   # shared ceiling (see §4.4)
)
\end{lstlisting}

\section{Tier 0.5: Plan Coherence}
\label{sec:plan-coherence}
% ─────────────────────────────────────────────────────────────────────────────

A \code{ToolRequest} can be perfectly well-formed JSON and still be the
wrong action --- requesting a tool the current goal has explicitly
excluded, or proposing a tool call when the active goal has already
transitioned to \code{COMPLETED}.  \component{GoalCoherencePlanValidator}
catches this class of error before the decision reaches dispatch.

\begin{lstlisting}[language=Python,
    caption={GoalCoherencePlanValidator.validate() (src/validators.py)},
    label={lst:plan-validator}]
class GoalCoherencePlanValidator:
    def validate(
        self, decision: Any, state: AgentState,
    ) -> tuple[str, str]:           # (status, reason)
        goals  = state.get("goals") or []
        active = [g for g in goals if g.status == GoalStatus.ACTIVE]

        if not active:
            return ("failed", "No ACTIVE goals found; cannot validate plan.")

        if isinstance(decision, FinalAnswer):
            return ("passed", "")   # always coherent — closes the loop

        if isinstance(decision, ToolRequest):
            for g in active:
                if decision.tool_name in g.excluded_tools:
                    return ("needs_correction",
                           f"Tool '{decision.tool_name}' excluded for "
                           f"goal '{g.description[:60]}'")

        return ("passed", "")
\end{lstlisting}
\githubref{src.validators.GoalCoherencePlanValidator}

The validator returns one of three statuses.
\textbf{\code{passed}}: proceed to \code{action\_node}.
\textbf{\code{needs\_correction}}: increment \code{revision\_count} and
retry --- the self-healing loop.
\textbf{\code{failed}}: terminate immediately with
\code{PLAN\_INCOHERENT}; no retry can fix an empty goal stack.

\begin{lstlisting}[language=Python,
    caption={Branching on plan-validation status (src/graph.py, excerpt)},
    label={lst:plan-branch}]
status, reason = plan_validator.validate(decision, state)
revision_count = state.get("revision_count") or 0

if status == "needs_correction":
    revision_count += 1
    if revision_count >= policy.max_retries:
        return {"run_state":          RunState.TERMINATED,
                "termination_reason": TerminationReason.PLAN_INCOHERENT,
                "revision_count":     revision_count}
    return {"revision_count": revision_count, "decision": None}

if status == "failed":
    return {"run_state":          RunState.TERMINATED,
            "termination_reason": TerminationReason.PLAN_INCOHERENT}
\end{lstlisting}
\githubref{src.graph.reason\_node}

\begin{warningbox}
\textbf{\code{needs\_correction} and \code{failed} must not share a
termination path.}  \code{needs\_correction} is correctable by retrying;
\code{failed} indicates a state bug that retrying cannot fix.  Routing
both to the same branch wastes the entire retry budget on an impossible
request.
\end{warningbox}

\section{The Repair Loop and Its Ceiling}
\label{sec:repair-loop}
% ─────────────────────────────────────────────────────────────────────────────

\begin{table}[htbp]
\centering
\caption{Post-validation routing table}
\label{tab:routing-table}
\renewcommand{\arraystretch}{1.4}
\begin{tabularx}{\linewidth}{@{} l l P{4.6cm} @{}}
\toprule
\textbf{Condition} & \textbf{Route} & \textbf{Why} \\
\midrule
Passed                                     & \code{action}
  & Dispatch through the 4-layer chain (Chapter~1 §\ref{sec:action-node}) \\
Failed, requires approval (Safety layer)   & \code{hitl}
  & Chapter~21 --- surface an \code{ApprovalPacket} \\
Failed, attempt $<$ \code{max\_retries}    & \code{reason}
  & Retry with corrective feedback \\
Failed, attempt $\geq$ \code{max\_retries} & \code{escalation}
  & Ceiling reached; write \code{dead\_letter} \\
\bottomrule
\end{tabularx}
\end{table}

\begin{notebox}
\textbf{One shared ceiling.}  \code{TerminationPolicy.max\_retries}
bounds Instructor's parse retries, the plan-coherence
\code{needs\_correction} loop, and Chapter~14's tool-dispatch retry
budget from one field.  An operator tuning ``how persistent should this
agent be'' changes one number, not three.
\end{notebox}

\section{Applying the Subgraph Principle to \texttt{reason\_node}}
\label{sec:ch4-subgraph}
% ─────────────────────────────────────────────────────────────────────────────

\code{reason\_node} now performs three visibly distinct steps: an
Instructor call, a plan-coherence check, and a budget drain (Chapter~5).
Applying the Subgraph Principle (Chapter~1 §\ref{sec:subgraph-principle})
directly:

\begin{takeawaybox}[Subgraph Principle applied: verdict — single node]
\begin{enumerate}
  \item \emph{Three or more steps?}  Yes --- model call, plan validation,
        budget drain.
  \item \emph{Distinct failure modes worth separate spans?}  Partially.
        A parse failure differs from a coherence failure, but neither
        is distinct enough from ``the model call failed'' to justify a
        separate recovery path --- a crash anywhere here means
        re-calling the model from the top.
  \item \emph{Communicate through \code{AgentState}, not local variables?}
        \textbf{No.}  \code{decision}, \code{status}, and
        \code{revision\_count} are local Python values inside one function
        body until the single \code{return} at the end.
  \item \emph{I/O boundary, or resumability value between steps?}
        \textbf{No.}  Plan validation and budget draining are
        synchronous, in-memory operations on data the model call already
        returned.  There is no scenario where resuming ``after the model
        call but before plan validation'' has operational value.
\end{enumerate}
Conditions~3 and~4 both fail.  \code{reason\_node} stays a single node.
This is the first of three consecutive chapters (4, 5, 6) where an
apparently multi-step function is tested honestly and kept flat.
\end{takeawaybox}

\begin{takeawaybox}[Chapter~4 Key Takeaways]
\begin{itemize}
  \item \textbf{Three validation tiers, three locations.}  Output-schema
        parsing and plan coherence both live inside \code{reason\_node}.
        Action validation lives in \code{action\_node}.

  \item \textbf{Instructor mode is a provider decision, not a constant.}
        \code{TOOLS} by default; \code{JSON} or \code{MD\_JSON} only
        when the provider genuinely lacks function-calling support.

  \item \textbf{\code{needs\_correction} and \code{failed} route
        differently.}  One is correctable by retrying; the other
        indicates a state bug that retrying cannot fix.

  \item \textbf{One shared retry ceiling.}  \code{TerminationPolicy.\allowbreak
        max\_retries} bounds parse retries, plan-coherence retries, and
        tool-dispatch retries from one field.

  \item \textbf{\code{reason\_node} correctly stays a single node.}
        Fails the Subgraph Principle on conditions~3 and~4 (local-variable
        communication; no resumability value between steps) even though it
        performs real network I/O.
\end{itemize}
\end{takeawaybox}

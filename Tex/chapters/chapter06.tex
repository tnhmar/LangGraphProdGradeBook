\chapterquote{%
  The context window is not a text box.  It is a budget, with line
  items, and an agent that does not account for every token in it is
  an agent spending money it has not counted.%
}{Engineering AI Agents for Production, Volume~1}

\begin{learningbox}
After completing this chapter you will be able to:
\begin{itemize}
  \item Allocate the context window into named, individually budgeted
        slots --- \term{ROLE}, \term{ACTIVE\_GOAL}, \term{TOOLS},
        \term{EVIDENCE}, \term{HISTORY} --- rather than concatenating
        text until the window is full.
  \item Enforce per-slot token budgets with \code{tiktoken}
        \emph{before} assembly, so an overflow is never discovered after
        the prompt has already been sent.
  \item Order slots for \term{KV-cache reuse}: stable content first,
        variable content last.
  \item Wire the \term{EVIDENCE} slot to memory-retrieval beliefs,
        closing a gap that left it empty in earlier designs.
  \item Confirm by direct application of the Subgraph Principle that a
        six-step prompt-assembly process correctly remains a single node
        --- the third consecutive chapter to reach this verdict.
\end{itemize}
\end{learningbox}

\section{The Context Window as a Typed Resource}
\label{sec:context-resource}
% ─────────────────────────────────────────────────────────────────────────────

Volume~1 Chapter~3 treats the context window as a scarce resource with an
explicit budget.  \component{SlotBudget} declares five named slots plus a
reserved headroom:

\begin{lstlisting}[language=Python,
    caption={SlotBudget --- named, individually budgeted slots
             (src/context.py)},
    label={lst:slot-budget}]
class SlotBudget(BaseModel):
    model_config = ConfigDict(frozen=True)
    role:        int = Field(default=500,  ge=1)
    active_goal: int = Field(default=300,  ge=1)
    tools:       int = Field(default=800,  ge=1)
    evidence:    int = Field(default=2000, ge=1)
    history:     int = Field(default=1500, ge=1)
    headroom:    int = Field(default=500,  ge=1)   # reserved for output

    EVICTION_ORDER: list[str] = ["history", "evidence", "tools"]
\end{lstlisting}
\githubref{src.context.SlotBudget}

\code{headroom} is subtracted from the total window before any slot is
allocated.  \code{EVICTION\_ORDER} declares which slots give up space
first under emergency pressure: history before evidence, evidence before
tools.  \code{role} and \code{active\_goal} are absent from the eviction
order --- they are never evicted.

% ─────────────────────────────────────────────────────────────────────────────
\section{Per-Slot Budget Enforcement}
\label{sec:budget-enforcement}
% ─────────────────────────────────────────────────────────────────────────────

Every slot is trimmed to its budget independently, before assembly.

\begin{lstlisting}[language=Python,
    caption={trim\_to\_budget --- per-slot enforcement (src/context.py)},
    label={lst:trim-to-budget}]
def count_tokens(text: str, encoder: tiktoken.Encoding) -> int:
    return len(encoder.encode(text))

def trim_to_budget(text: str, budget: int,
                   encoder: tiktoken.Encoding) -> str:
    tokens = encoder.encode(text)
    if len(tokens) <= budget:
        return text
    return encoder.decode(tokens[:budget]) + " [\u2026TRIMMED]"
\end{lstlisting}
\githubref{src.context.trim\_to\_budget}

\begin{warningbox}
\textbf{Trim before assembly, never after.}  Trimming the fully assembled
prompt string has no way to know which slot contributed the excess tokens.
It risks truncating mid-slot --- half of a JSON tool description, for
instance.  Each slot is independently budget-compliant before it is
concatenated with any other.
\end{warningbox}

\section{The Goal Summary: Re-Injected Every Cycle}
\label{sec:goal-summary}
% ─────────────────────────────────────────────────────────────────────────────

The \code{ACTIVE\_GOAL} slot is rebuilt from \code{AgentState} on every
cycle so the model always sees the agent's \emph{current} understanding of
progress, not a stale copy from cycle~1.

\begin{lstlisting}[language=Python,
    caption={GoalSummary --- compact, re-rendered every cycle
             (src/context.py)},
    label={lst:goal-summary}]
class GoalSummary(BaseModel):
    model_config = ConfigDict(frozen=True)
    original_goal: str
    current_step:  str
    constraints:   list[str] = Field(default_factory=list)
    priority:      str       = "accuracy"
    progress_pct:  float     = Field(default=0.0, ge=0.0, le=100.0)

    def render(self) -> str:
        lines = [
            f"GOAL: {self.original_goal}",
            f"STEP: {self.current_step}",
            f"PROGRESS: {self.progress_pct:.0f}%",
        ]
        if self.constraints:
            lines.append("CONSTRAINTS: " + "; ".join(self.constraints))
        return "\n".join(lines)
\end{lstlisting}
\githubref{src.context.GoalSummary}

\code{current\_step} is read from the first \code{ACTIVE} sub-goal in
the hierarchy (Chapter~3 §\ref{subsec:goals}), falling back to the root
goal text when no sub-goal has been activated yet.  The
\code{CONSTRAINTS} line appears only when constraints exist --- no empty
label is emitted every cycle.

\section{The EVIDENCE Slot: Closing the Memory-Injection Gap}
\label{sec:evidence-slot}
% ─────────────────────────────────────────────────────────────────────────────

\code{context\_assembly\_node} populates the \code{EVIDENCE} slot from
two sources: beliefs whose \code{provenance} field indicates they are
memory-derived, and \code{ACTIVE}-tier working-memory entries.

\begin{lstlisting}[language=Python,
    caption={EVIDENCE slot population (src/context.py, excerpt)},
    label={lst:evidence-slot}]
beliefs = state.get("beliefs") or []
wm      = state.get("working_memory") or []

evidence_parts = []
for b in beliefs[-30:]:
    if "memory" in b.provenance or "retrieval" in b.provenance:
        evidence_parts.append(
            f"Memory [{b.provenance} {b.asserted_at[:10]}]: {b.object_}"
        )
for e in wm:
    if e.tier == WorkingMemoryTier.ACTIVE:
        evidence_parts.append(f"Context: {e.content[:200]}")

evidence_text = trim_to_budget(
    "\n".join(evidence_parts[-20:]), budget.evidence, encoder
)
\end{lstlisting}
\githubref{src.context.context\_assembly\_node}

Two upstream nodes write the beliefs this slot consumes.
\code{grounding\_node} in the PerceiveSubgraph (Chapter~1) tags its
assertions with \code{provenance="grounding\_node"}.
\code{memory\_read\_node} (Chapter~8) tags its retrieval results with
\code{provenance="memory\_read\_node"}.  Filtering on a substring match
against \code{provenance} means any future node wanting its assertions
surfaced as context evidence only needs to choose a provenance string
containing \code{"memory"} or \code{"retrieval"} --- no change to
\code{context\_assembly\_node} is required.

\begin{tipbox}
\textbf{\code{ACTIVE}-tier working memory bypasses the relevance cap.}
Disambiguation entries written by \code{stuck\_recovery\_node}
(Chapter~2) are tagged \code{WorkingMemoryTier.ACTIVE} specifically so
they reach this slot on the very next cycle, without competing against
the 30-entry belief window or 20-entry evidence cap.
\end{tipbox}

\section{KV-Cache-Friendly Slot Ordering}
\label{sec:kv-cache}
% ─────────────────────────────────────────────────────────────────────────────

Most inference providers cache the key-value attention state for a
prompt's prefix and reuse it when that prefix is byte-identical.  Slot
order determines how much of each request's compute is new work.

\begin{lstlisting}[language=Python,
    caption={Slot assembly order (src/context.py, excerpt)},
    label={lst:slot-order}]
assembled = [
    {"role": "system", "content": role_text},        # ROLE   — most stable
    {"role": "system", "content": goal_text},         # GOAL   — slow-changing
]
if tools_text:
    assembled.append({"role": "system", "content": tools_text})
if evidence_text:
    assembled.append({"role": "system", "content": evidence_text})
if history_text:
    assembled.append({"role": "system", "content": history_text})  # most volatile
\end{lstlisting}
\githubref{src.context.context\_assembly\_node}

\code{role\_text} is identical on every cycle of every run using the
same configuration --- the most stable possible prefix, placed first.
\code{evidence} and \code{history} change on essentially every cycle and
trail it, where their volatility cannot invalidate the cache for the more
stable prefix.

\section{Emergency Eviction}
\label{sec:emergency-eviction}
% ─────────────────────────────────────────────────────────────────────────────

After per-slot trimming, the slot totals can still together exceed the
available window.  A final check applies a second trim only to the lowest-
priority slots.

\begin{lstlisting}[language=Python,
    caption={Emergency eviction on total overflow (src/context.py, excerpt)},
    label={lst:emergency-eviction}]
total = sum(count_tokens(t, encoder) for t in
            [role_text, goal_text, tools_text,
             evidence_text, history_text] if t)
available = context_window - budget.headroom

if total > available:
    history_text  = trim_to_budget(
        history_text,  int(budget.history  * 0.5), encoder)
    evidence_text = trim_to_budget(
        evidence_text, int(budget.evidence * 0.5), encoder)
\end{lstlisting}
\githubref{src.context.context\_assembly\_node}

\code{role\_text} and \code{goal\_text} are absent from emergency
eviction.  An agent whose role and current goal cannot fit after history
and evidence are halved has a configuration problem, not a content
problem, and should fail loudly rather than silently truncate its own
identity.

\section{Applying the Subgraph Principle to \texttt{context\_assembly\_node}}
\label{sec:ch6-subgraph}
% ─────────────────────────────────────────────────────────────────────────────

Six named steps build this prompt: role, goal summary, tools, evidence,
history, and the final budget check.  The highest step count of any
construct examined in this book so far.  The Subgraph Principle
(Chapter~1 §\ref{sec:subgraph-principle}) exists precisely to prevent
pattern-matching on step count.

\begin{takeawaybox}[Subgraph Principle applied: verdict — single node]
\begin{enumerate}
  \item \emph{Three or more steps?}  Yes --- six.
  \item \emph{Distinct failure modes worth separate spans?}
        \textbf{No.}  Every slot has exactly the same failure mode: this
        slot exceeded its budget and was trimmed.  The slot name differs;
        the operational meaning of the event does not.  Contrast
        PerceiveSubgraph, where a PII leak, a hostile injection, and a
        relevance filter failure are three categorically different
        incidents investigated differently.
  \item \emph{Communicate through \code{AgentState}, not local variables?}
        \textbf{No, decisively.}  \code{role\_text},
        \code{goal\_text}, \code{tools\_text}, \code{evidence\_text},
        and \code{history\_text} are local Python variables inside one
        function body, combined into one list at the end.  Nothing about
        an individual slot exists independently in \code{AgentState}.
  \item \emph{I/O boundary, or resumability value between steps?}
        \textbf{No, on both counts.}  Every step is local string and
        token-count arithmetic.  A crash anywhere here means
        re-running the function from the top, which costs microseconds.
\end{enumerate}
Three of four conditions fail.  This is the cleanest negative verdict in
the book: six steps and zero I/O is the textbook single-node shape.
Conditions~2, 3, and~4 failing independently gives three separate
reasons --- any one of them would be sufficient.
\end{takeawaybox}

def test_evidence_includes_memory_tagged_beliefs():
    beliefs = [
        Belief(subject="s", predicate="p", object_="result A",
              provenance="memory_read_node"),
        Belief(subject="s", predicate="p", object_="irrelevant",
              provenance="grounding_node"),

    assert slots[0] == "ROLE"

    long_text = " ".join(f"w{i}" for i in range(20))
    trimmed   = trim_to_budget(long_text, 5, encoder)
    assert trimmed.endswith("\u2026TRIMMED]")

\begin{takeawaybox}[Chapter~6 Key Takeaways]
\begin{itemize}
  \item \textbf{Named slots, not concatenation.}  \code{SlotBudget} gives
        ROLE, ACTIVE\_GOAL, TOOLS, EVIDENCE, and HISTORY independent token
        ceilings with an explicit eviction order.

  \item \textbf{Trim every slot before assembly, never the assembled
        prompt afterward.}

  \item \textbf{The goal summary is rebuilt every cycle,} keeping
        progress, constraints, and the current step current.

  \item \textbf{EVIDENCE is wired to provenance, not hardcoded sources.}
        Any node tagging beliefs with a provenance containing
        \code{"memory"} or \code{"retrieval"} automatically surfaces here.

  \item \textbf{Stable content first, volatile content last} for
        KV-cache prefix reuse.

  \item \textbf{Six steps, zero I/O, three failing conditions: the
        definitive single-node example.}  Chapters~4, 5, and~6 each
        independently confirmed that the Subgraph Principle produces
        ``no'' as often as ``yes'', and that step count does not drive
        the answer.
\end{itemize}
\end{takeawaybox}
